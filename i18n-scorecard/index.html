i18n / cjk readiness scorecard

Point it at a repo. It scans for the text bugs that quietly break Japanese, Korean, Chinese, emoji, and right-to-left input — then grades the repo out of 100, with a shareable card, a README badge, and an exit code you can gate CI on.

Run it

$ npx @greymoth/i18n-scorecard ./your-repo

Writes a scorecard, a 1200×630 share card, and a badge into ./i18n-report/. No config, no account, nothing leaves your machine.

See a sample scorecard → The share card

What it looks for

  • Grapheme miscounts — a .length or .substring that splits a kanji or a flag emoji in half.
  • IME-submit races — an Enter handler that fires mid-conversion, submitting half-typed Japanese.
  • Homoglyphs & invisibles — Cyrillic а posing as Latin a, zero-width characters splicing a token.
  • Locale drift — a translation file silently falling behind its English source.
  • Encoding & width — UTF-8 read as Latin-1, fullwidth forms surviving normalization into something dangerous.

Thirteen checks across seven categories. The detection is lifted straight from a year of real fixes — catalogued here — not written from a checklist.

Badge it

drops into a README from badge.md
The score is a hazard reading, not a coverage grade: a repo with no CJK, emoji, or RTL text scans clean and scores 100 because it has nothing to get wrong. It earns its keep on codebases that actually handle international text.

Where it comes from

Built on Cursed Strings, a field guide to real text bugs, and a public corpus of the CJK/Unicode ones — each linked to an actual pull request, not remembered.
